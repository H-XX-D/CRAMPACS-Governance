# CRAMPACS Brand System

**Document ID:** CRAMPACS-BRAND-001  
**Version:** 0.1  
**Status:** Draft brand standard  
**Owner:** CRAMPACS program owner

## 1. Brand Position

CRAMPACS is a disciplined assurance method for testing whether weak, heterogeneous signals recur at pre-specified coordinates more often than expected under controlled null models.

It should feel:

- serious
- inspectable
- teachable
- operational
- statistically cautious
- useful to supervisors and practitioners

It should not feel:

- speculative
- mystical
- promotional
- discovery-claiming
- jargon-heavy without controls
- like a loose collection of templates

## 2. Brand Promise

CRAMPACS turns weak-signal pattern suspicion into a controlled evidence package.

## 3. Brand Boundary

CRAMPACS is not a proof engine. It is an evidence-control and anomaly-prioritization system.

Permitted claim:

> A pre-specified coordinate shows unusual recurrence under the registered CRAMPACS method and should be prioritized for follow-up.

Prohibited claim:

> CRAMPACS proves the underlying causal, clinical, financial, security, engineering, or scientific conclusion.

## 4. Naming System

| Name | Meaning | Use |
|---|---|---|
| `crampacs-*` | lightweight preflight | 1 to 2 day triage and escalation decision |
| `CRAMPACS-*` | full assurance system | locked protocol, complete evidence package, review and release |

Examples:

- `crampacs-med` is the medicine preflight.
- `CRAMPACS-MED` is the full medicine assurance system.
- `crampacs-fin` is the finance preflight.
- `CRAMPACS-FIN` is the full finance assurance system.

Never use uppercase for the lightweight route.

## 5. Verbal Identity

Preferred language:

- coordinate recurrence
- evidence package
- protocol lock
- null and non-event evidence
- dependence map
- bias register
- sidecar blockers
- assurance tier
- release hold
- supervisor decision

Avoid:

- proof
- discovery
- signal mining
- magic meta-analysis
- weak evidence made strong
- automated truth
- regulatory compliance by default

## 6. Visual Identity

CRAMPACS should visually read as a controlled technical system.

### Color Roles

These colors are guidance for future documents, web pages, decks, and platform UI.

| Role | Hex | Use |
|---|---|---|
| Ink | `#17212B` | body text, titles |
| Slate | `#44515F` | secondary text, table labels |
| Signal Blue | `#1E6B9E` | links, active states, coordinate references |
| Evidence Green | `#2F7D5A` | pass, accepted evidence, verified checks |
| Hold Amber | `#B36B00` | hold, rework, warning |
| Stop Red | `#A83A32` | critical finding, release stop |
| Paper | `#F7F8FA` | document background |
| Line | `#D8DEE6` | rules, table borders |

### Typography

Use plain, readable sans-serif type for operational material.

Recommended stack:

```text
Inter, Aptos, Segoe UI, Helvetica Neue, Arial, sans-serif
```

For code, IDs, hashes, paths, and commands:

```text
SFMono-Regular, Consolas, Liberation Mono, Menlo, monospace
```

## 7. Document Badge System

Use these badges near the top of controlled docs:

| Badge | Meaning |
|---|---|
| `Level: crampacs-*` | lightweight preflight only |
| `Level: CRAMPACS-*` | full assurance system |
| `Gate: G0-G7` | gate controlled by canonical gate map |
| `Status: Draft` | not approved for controlled use |
| `Status: Approved` | approved under document control |
| `Release: Hold` | cannot be externally used |
| `Release: Approved` | approved only within stated evidence tier |

## 8. Document Header Pattern

Every controlled document should start with:

```text
# Document Title

**Document ID:** CRAMPACS-XXX-001
**Version:** 0.1
**Status:** Draft
**Owner:** 
**Approver:** 
**Effective date:** 
**Review due:** 
**Applies to:** crampacs-* / CRAMPACS-* / both
```

## 9. Layout Pattern

Use this order for practitioner-facing documents:

1. Purpose
2. When to use
3. Inputs
4. Procedure
5. Gates or decisions
6. Required evidence
7. Stop rules
8. Outputs
9. Review or signoff

## 10. Brand Quality Bar

A CRAMPACS artifact is brand-complete only if:

- the assurance level is explicit
- the claim boundary is visible
- inputs and outputs are clear
- stop rules are easy to find
- evidence requirements are concrete
- the reader knows what to do next
- the document avoids unsupported domain claims
